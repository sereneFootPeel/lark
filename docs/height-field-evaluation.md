# 二维栅格流量 / 高度场替代粒子核评估

## 结论

对当前 `lark` 项目而言，**建议把二维栅格流量/高度场作为可选的移动端极限省电后端，而不是立即替换现有粒子核**。

原因很直接：

- 现有验收指标 `topFraction`、`throatOccupancy`、`maxBoundaryViolation`、`isolatedFastParticleCount` 都天然围绕粒子分布设计。
- 现在的显示层已经是“粒子 -> 显示点阵/占据场”的映射，这意味着粒子核仍能直接复用显示系统。
- 高度场/流量场在矩形容器里很强，但沙漏这种**细颈、双腔、斜边界、局部堵塞/松散再流动**的视觉语义，单纯高度场会显著损失“颗粒感”和局部随机性。

## 适合替代的场景

二维栅格方案更适合下面这些目标：

1. **极限省电档**
   - 长时间待机动画
   - 低端 Android 机型
   - 需要把 CPU 占用压到最低

2. **视觉优先于局部流体真实性**
   - 只要求“上面变少、下面变多、颈口有流动”
   - 不要求粒子喷溅、局部拥堵、黏连等微观细节

3. **显示层本身已经是离散点阵**
   - 当前 `HourglassDisplayField` 非常适合接栅格占据率
   - 不需要保留逐粒子的渲染身份

## 不适合立即替代的原因

### 1. 沙漏几何不是规则水槽

普通高度场通常假设：

- 每列有单值高度
- 流动主要沿重力方向
- 横向耦合较弱

但沙漏在 `y=0` 附近存在：

- 极窄颈口
- 上下双腔切换
- 45° 斜边
- 局部堆积与再松散

这会让单值高度场很快失真。

### 2. 视觉颗粒感会下降

当前粒子核虽然已经是低成本 SPH/位置松弛风格，但仍保留了：

- 局部随机分布
- 细颈处的“散粒流”
- 倾斜时的侧向偏置
- 局部快慢差异

如果直接改为高度场，最容易出现：

- 流动像“液柱”而不是“沙/液滴群”
- 颈口过于平滑
- 左右倾斜时局部堆积不够自然

### 3. 现有测试体系要重写一部分

一旦内核从粒子改为格子：

- `isolatedFastParticleCount` 失去意义
- `throatOccupancy` 需要改成格点占据率
- `centerOfMass()` 要从质量场积分计算

所以替代内核更像是一个**并行实现**，不是“小改动”。

## 推荐的折中方案

### 方案 A：保留粒子核作为默认/平衡档

- `DESKTOP`：粒子核
- `MOBILE_BALANCED`：粒子核 + 当前 float + 自适应/像素缓冲渲染
- `MOBILE_BATTERY_SAVER`：粒子核低迭代，必要时再切格子核

这是当前最稳妥的路线。

### 方案 B：增加独立的 `GridFlowSimulation`

建议不要直接删掉 `HourglassSimulation`，而是新增：

- `GridFlowSimulation`
- 统一接口，例如 `FluidSimulationView`
- `LarkController.QualityProfile` 决定使用哪种内核

这样可以：

- 继续复用 `HourglassDisplayField`
- 并行对比视觉效果
- A/B 测试性能与验收指标

## 一个可行的栅格方案

### 数据表示

按规则网格离散沙漏内部：

- `mass[cell]`：当前质量/占据率
- `velX[cell]`, `velY[cell]`：可选，若要保留惯性
- `solid[cell]`：是否是边界外/玻璃区域
- `capacity[cell]`：该格最大容纳量，颈口可更低

### 每步更新

1. **重力投影**
   - 根据当前重力方向计算目标流向

2. **局部通量**
   - 在 4 邻域或 8 邻域上计算质量转移
   - 对颈口附近通量施加上限，模拟细口限流

3. **质量守恒修正**
   - 保证转出 <= 当前质量
   - 保证转入不超过容量

4. **平滑/扩散**
   - 少量横向扩散，避免完全“方块化”

5. **显示映射**
   - 把 `mass` 双线性采样到现有显示点阵或直接采样到像素缓冲

## 预估收益

如果切到格子核，理论收益主要来自：

- 邻域搜索消失
- 复杂度从“粒子数 × 邻域”转为“格子数 × 固定邻居”
- 内存连续性更好

在移动端通常可能获得：

- CPU 降低约 2x 到 6x（取决于格子分辨率）
- 更稳定的帧时间
- 更少的随机抖动

## 主要风险

1. **视觉风格改变明显**
2. **倾斜时侧向堆积不如粒子自然**
3. **颈口流动会更像黏性体积流**
4. **测试与调参成本不小**

## 推荐落地顺序

1. 先继续打磨当前粒子核
   - 已完成 float 化
   - 已完成像素缓冲渲染
   - 已加入省电档自适应迭代

2. 如果仍不满足移动端预算，再新增 `GridFlowSimulation`
   - 先只服务 `MOBILE_BATTERY_SAVER`
   - 保持 `MOBILE_BALANCED` 继续使用粒子核

3. 新增一组并行验收指标
   - 上下腔质量比例
   - 颈口区域占据率时间曲线
   - 质心轨迹
   - 平均 CPU 时间/step

## 最终建议

**短期：不要替换默认粒子核。**

**中期：把二维栅格流量/高度场实现成 `MOBILE_BATTERY_SAVER` 的候选后端。**

这样能同时满足：

- 保住当前视觉风格
- 在低端手机上继续向下扩性能
- 不破坏现有测试与渲染点阵体系
